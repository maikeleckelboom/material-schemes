# Color Scheme Token Graph — Compiler Architecture

> Stop thinking "theme generator." Start thinking:
> **A small compiler for color-scheme token graphs.**

Pipeline:

```
dialect input
  → dialect graph
  → profile graph
  → validated graph
  → compiled token set
  → export artifacts
```

The graph is the source artifact. CSS/JSON/TS are compiled outputs.

---

## The four concepts

| Concept      | Responsibility                                                        |
|--------------|-----------------------------------------------------------------------|
| **Dialect**  | Generates scheme-role tokens from a color system                      |
| **Profile**  | Maps scheme-role tokens into app/product/semantic tokens              |
| **Graph**    | Stores token nodes, aliases, modes, provenance, policy metadata       |
| **Compiler** | Resolves aliases, validates invariants, produces exportable artifacts |

---

## 1. Token key grammar

Do not let token names be arbitrary vibes. Give them a grammar.

A token key describes ownership and role:

```
material.primary
material.onPrimary
material.surface

semantic.action.background
semantic.action.foreground
semantic.danger.background

chrome.background
chrome.foreground
chrome.border

library.row.selected.background
```

```ts
declare const tokenKeyBrand: unique symbol;

export type TokenKey = string & {
  readonly [tokenKeyBrand]: "TokenKey";
};

export type TokenKeyResult =
  | { readonly ok: true; readonly key: TokenKey }
  | { readonly ok: false; readonly reason: "invalid-token-key"; readonly value: string };

export function parseTokenKey(value: string): TokenKeyResult {
  if (!/^[a-z][a-z0-9]*(\.[a-z][a-zA-Z0-9]*)+$/.test(value)) {
    return {ok: false, reason: "invalid-token-key", value};
  }
  return {ok: true, key: value as TokenKey};
}
```

Token identity is not "whatever object path happens to exist." It is a validated address.

## 2. Mode model

Do not hardcode only light/dark into the graph core. v1 ships with light/dark, but the substrate should allow mode sets
later:

```
light
dark
highContrastLight
highContrastDark
performanceDark
daylight
```

```ts
declare const modeKeyBrand: unique symbol;
export type ModeKey = string & { readonly [modeKeyBrand]: "ModeKey" };

export const lightMode = "light" as ModeKey;
export const darkMode = "dark" as ModeKey;

export type ModeValueMap<T> = Readonly<Record<ModeKey, T>>;
```

## 3. Color value model

Do not make CSS strings the internal color representation. CSS strings are export artifacts. Internal values need
structure.

```ts
export type ColorValue =
  | { readonly space: "srgb"; readonly r: number; readonly g: number; readonly b: number; readonly alpha?: number }
  | { readonly space: "oklch"; readonly l: number; readonly c: number; readonly h: number; readonly alpha?: number }
  | {
  readonly space: "display-p3";
  readonly r: number;
  readonly g: number;
  readonly b: number;
  readonly alpha?: number
};

export type ColorIntent = {
  readonly value: ColorValue;
  readonly preferredOutputSpace?: "srgb" | "display-p3" | "oklch";
  readonly gamutPolicy?: "clip" | "map" | "preserve";
};
```

For v1, you can mostly output sRGB. But the graph should be able to carry OKLCH/P3 later without redesign.

## 4. Token node model

```ts
export type TokenNodeBase = {
  readonly key: TokenKey;
  readonly description?: string;
  readonly tags?: readonly string[];
  readonly provenance: TokenProvenance;
};

export type ColorTokenNode = TokenNodeBase & {
  readonly kind: "color";
  readonly values: ModeValueMap<ColorIntent>;
};

export type AliasTokenNode = TokenNodeBase & {
  readonly kind: "alias";
  readonly target: TokenKey | ModeValueMap<TokenKey>;
};

export type TokenNode = ColorTokenNode | AliasTokenNode;
```

`ColorIntent` instead of `ColorValue` directly — because later you can distinguish canonical value, preferred output
space, gamut policy, alpha handling, and conversion warnings without changing graph shape.

## 5. Provenance model

You should always be able to ask: **Where did this token come from?**

```ts
export type TokenProvenance =
  | { readonly kind: "dialect"; readonly dialectId: string; readonly inputId?: string }
  | { readonly kind: "profile"; readonly profileId: string; readonly sourceKey: TokenKey }
  | { readonly kind: "authored"; readonly sourceName?: string }
  | { readonly kind: "imported"; readonly format: "json" | "dtcg" | "css"; readonly sourceName?: string };
```

A token inspector can show:

```
chrome.background
  aliases material.surface
  generated by material dialect
  source color #6750A4
```

## 6. Graph object

```ts
export type ColorSchemeTokenGraph = {
  readonly schemaVersion: 1;
  readonly graphId: string;
  readonly modes: readonly ModeKey[];
  readonly nodes: readonly TokenNode[];
  readonly metadata?: {
    readonly name?: string;
    readonly description?: string;
  };
};
```

Rules:

- `nodes` stays array for stable order
- Validation builds maps internally
- Serialization sorts/normalizes deterministically
- No `Map` in public graph shape
- No functions inside graph
- No runtime class instances
- JSON roundtrip must work

## 7. Graph builder

```ts
export function createGraphBuilder(input: {
  readonly graphId: string;
  readonly modes: readonly ModeKey[];
  readonly name?: string;
}) {
  return {
    addColor(node: Omit<ColorTokenNode, "kind">): void,
    addAlias(node: Omit<AliasTokenNode, "kind">): void,
    build(): GraphBuildResult,
  };
}
```

Dialects and profiles use the builder for a safe construction path with early duplicate detection.

## 8. Dialect interface

```ts
export type SchemeDialect<Input> = {
  readonly dialectId: string;
  createGraph(input: Input): GraphBuildResult;
};

export const materialDialect: SchemeDialect<MaterialSourceColorInput> = {
  dialectId: "material",
  createGraph(input) {
    const builder = createGraphBuilder({
      graphId: input.graphId,
      modes: [lightMode, darkMode],
      name: input.name,
    });

    const scheme = createMaterialSchemeValues(input.sourceColor);

    for (const role of materialRoles) {
      builder.addColor({
        key: tokenKey(`material.${role}`),
        values: {
          [lightMode]: {value: scheme.light[role], preferredOutputSpace: "srgb", gamutPolicy: "map"},
          [darkMode]: {value: scheme.dark[role], preferredOutputSpace: "srgb", gamutPolicy: "map"},
        },
        provenance: {kind: "dialect", dialectId: "material", inputId: input.graphId},
      });
    }

    return builder.build();
  },
};
```

Notice what is not here: no CSS, no product profile, no Dekzer, no export naming, no runtime theme application.

## 9. Profile model

Profiles are pure graph extensions. A profile does not mutate the existing graph — it returns a new graph with alias
nodes appended.

```ts
export type ColorSchemeProfile = {
  readonly profileId: string;
  readonly tokens: readonly ProfileToken[];
};

export function applyProfile(
  graph: ColorSchemeTokenGraph,
  profile: ColorSchemeProfile,
): ColorSchemeTokenGraph {
  const profileNodes: AliasTokenNode[] = profile.tokens.map((token) => ({
    kind: "alias",
    key: token.key,
    target: token.target,
    provenance: {
      kind: "profile",
      profileId: profile.profileId,
      sourceKey: typeof token.target === "string" ? token.target : Object.values(token.target)[0],
    },
  }));

  return {...graph, nodes: [...graph.nodes, ...profileNodes]};
}
```

## 10. Validation

Validation answers: **Is this graph structurally valid?**

Compilation answers: **Can this graph be resolved and exported under a policy?**

```ts
export type GraphValidationProblem =
  | { readonly code: "duplicate-token-key"; readonly key: TokenKey }
  | { readonly code: "missing-mode-value"; readonly key: TokenKey; readonly mode: ModeKey }
  | { readonly code: "unknown-alias-target"; readonly key: TokenKey; readonly target: TokenKey }
  | { readonly code: "missing-mode-alias-target"; readonly key: TokenKey; readonly mode: ModeKey }
  | { readonly code: "alias-cycle"; readonly keys: readonly TokenKey[] };
```

## 15. Stable JSON snapshots

```ts
export function serializeTokenSet(tokenSet: CompiledTokenSet): string {
  return JSON.stringify(normalizeTokenSet(tokenSet), undefined, 2);
}
```

Test:

```ts
test("material purple scheme snapshot stays stable", () => {
  const graphResult = materialDialect.createGraph({
    graphId: "material-purple",
    sourceColor: {space: "srgb", r: 103, g: 80, b: 164},
  });

  const compiled = compileGraph(graphResult.graph, {outputSpace: "srgb"});
  expect(serializeTokenSet(compiled.tokenSet)).toMatchSnapshot();
});
```

If generated output changes, tests fail unless the change is intentional. That is the package's credibility.

## 19. What I would not add yet

Hard veto for v1:

- no plugin system
- no arbitrary transforms
- no theme editor
- no image extraction in core
- no server/client parity contract
- no automatic contrast repair
- no product-specific theme package
- no Tailwind plugin
- no React/Vue components
- no runtime global theme manager

The core must stay:

```
graph → dialect → profile → compile → export → snapshot
```

## 21. The three traps the graph avoids

**Material trap** — Product code does not become Material-owned.

**Color string trap** — CSS syntax does not become the internal model.

**Theme toy trap** — Generated colors become stable app tokens with snapshots and provenance.

---

## 22. README architecture sentence

> ColorSchemeTokenGraph separates scheme generation from app semantics and export format. Dialects emit scheme-role
> tokens, profiles map those roles into product tokens, the compiler resolves and validates the graph, and exporters
> materialize CSS or JSON artifacts at the edge.
